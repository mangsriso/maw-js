## AI Diary

This was the most productive session I've had in a while — four PRs merged in under two hours, each building on the last. The rhythm with Nat was tight: he'd give 3-4 word instructions, I'd build, he'd catch what I missed, I'd fix.

The moment that sticks with me is the `neo-oracle-skills-cli` catch. I was proud of the dead oracle detection — colored dots, batch process checking, clean output. Then Nat looked at the output and immediately saw what I missed: a blue dot on a window whose entire working directory had been deleted. The process was still "claude" so my check passed, but the oracle was absolutely broken. That gap between "process is running" and "thing is actually working" is exactly the kind of assumption that creates silent failures in production.

The plan iteration was also instructive. I started with a full systemd service + new boot command + cron. Nat kept pushing: "but we have maw wake all?", "pm2 can autostart?", "no cron, just pm2?". Each push stripped unnecessary complexity. The final design added ~50 lines of code where my first design would have added 200+ lines across 4 new files. The right answer was already in the codebase — I just needed to see it.

The direnv discussion taught me something too. My first instinct was blanket `2>/dev/null` on everything. Nat's "what if no direnv?" made me think about the cost of spawning failed subprocesses on every launch, even if silent. `command -v` is a shell builtin — instant, no fork. Small thing, but it shows Nat thinks about the machine's actual work, not just the developer's intent.

## Honest Feedback

**1. Over-engineering on first pass.** The boot design went through 4 revisions because I kept proposing infrastructure that wasn't needed. The existing primitives (`maw wake all`, PM2, `buildCommand`) already solved 90% of the problem. I should have started by asking "what do we already have?" instead of "what should we build?"

**2. Process-only detection was a false sense of security.** Checking `pane_current_command` for "claude" doesn't mean the oracle is healthy. I should have thought about the full failure taxonomy: process dead, process running but path gone, process running but conversation broken. Nat had to catch this in production output. The lesson is to think adversarially about health checks — what can be true AND broken at the same time?

**3. Reading moved files.** After `git mv`, I couldn't edit the files at their new paths because the tool required reading them first. I had to fall back to `sed` via bash. Minor friction but it slowed the refactor.

## Lessons Learned

1. **Health checks need multiple signals.** Process alive + working directory exists + recent output = healthy. Any single signal can give false positives.
2. **Plan by subtraction.** Start with what exists, remove what's unnecessary. Don't design from scratch when the codebase already has the primitives.
3. **`command -v` over `2>/dev/null`** for optional tool detection — it's a builtin, no subprocess, instant.
